## Round 3 — adversarial validation (gate batch + critic + root-cause find)

### verify-install.sh + Makefile restructure

`scripts/verify-install.sh` built as a **HOME-injectable home-state gate**: `XBREED_HOME` override lets fixture runs operate against a synthetic home tree without touching real `$HOME`. Four check classes:

| check_class | what it catches |
|---|---|
| `check_agents` | `~/.claude/agents/*.md` symlinks resolve under `$REPO_ROOT/templates/agents/`; missing or broken links flagged |
| `check_commands` | `~/.claude/commands/{wwkd,xb,xbgst,...}.md` symlinks resolve under `$REPO_ROOT/commands/`; `references/xbreed-shared.md` reference link |
| `check_dispatch` | `~/.local/templates/dispatch/*.md` file-for-file content parity with `templates/dispatch/` |
| `check_binaries` | `xbreed` + `xask` present at `~/.local/bin/`, executable, and `command -v` resolves to them (PATH-shadow check — skipped in fixture mode) |

Makefile restructured to `.ONESHELL` + `set -euo pipefail` globally, and split into:
- `verify-repo` — git-tree-only (cargo + shell tests, no home-dir reads)
- `verify-install` — home-state gate (`verify-install.sh` + `axis_family_schema_check.sh` + `xask_template_missing_fail_loud.sh` + `xask_thinking_budget_reachable.sh`)
- `verify` — `verify-repo verify-install` (the full gate)
- `install` — now runs `$(MAKE) verify-install` as an **unconditional atomic post-deploy gate** after every install

**verify-install caught real drift twice during its own build round:** while iterating on the check logic, `make install` fired `verify-install` and surfaced (a) a stale dispatch template copy and (b) a dangling agent symlink that had been left from a prior scratch test — both corrected before the milestone closed.

### ROOT CAUSE FIND — axis_family_schema_check.sh symlink-break on every run

**15:37/15:38 incidents:** executor.md appeared modified on two separate runs during the session. Root cause located: `tests/axis_family_schema_check.sh` was using `cp "$BAK" "$TARGET"` in its `restore()` trap. On Linux, `sed -i` breaks symlinks by creating a new regular file at `$TARGET` (rename-based replacement); `mv "$BAK" "$TARGET"` and `cp "$BAK" "$TARGET"` both restore *content* but NOT the link form — `executor.md` was silently converted from a symlink to a regular file on every mutation cycle.

**Fix:** `axis_family_schema_check.sh` now records `TARGET_LINK="$(readlink "$TARGET")"` before the mutation and, in `restore()`, checks whether the original was a symlink. If so, it re-creates the symlink via `ln -sfn "$TARGET_LINK" "$TARGET"` rather than copying the backup — content was never mutated in the link target, so no copy is needed. The file: `tests/axis_family_schema_check.sh` lines 22–43.

## Rejected alternatives surveyed

- **Static expectation matrix** (critic verdict a) — a hardcoded `EXPECTED=(...)` array inside verify-routing.sh would become a second SSoT and drift on the next lane change. Rejected: all checks derived from SSoT extraction at runtime.
- **14 per-lane scripts** (planner escalation flag) — 14×~50-line scripts for each xask lane. Rejected in R1: maintenance scales linearly; parameterized loop in ~200 lines replaces all 14.
- **`git checkout` for restore** (routing_drift_mutation.sh idiom question) — `git checkout` restores HEAD and silently destroys uncommitted working-tree fixes. Rejected (bit the judge twice during R2); `cp/mv` backup + explicit restore is the correct idiom.
- **`~/.claude/agents/` as fourth surface for verify-routing.sh** — excluded per plan scope; `verify-install.sh` symlink-liveness checks cover the home surface separately without duplicating the SSoT-extraction logic.
- **Per-surface check scripts** (connector second-order proposal) — would require N scripts to stay in sync with each other. Rejected: single-script multi-surface loop is the correct shape.
