**Claim:** Without `set -T` (functrace), the DEBUG trap does not fire inside shell functions, command substitutions `$(...)`, or subshell groups `(...)`. If the actual CC spawn command is wrapped in any of these, the trap misfires silently.

**Evidence (A3 critique, FM-B/C/D):**
> "FM-B: no `set -T` — DEBUG trap does NOT enter functions. Gate: `unset CLAUDE_CODE_EFFORT_LEVEL; f(){ : --agent-name ccs-scribe-x --team-name t; }; f; printf '%s\n' "${CLAUDE_CODE_EFFORT_LEVEL:-UNSET}"` → `UNSET` (trap never fired)."
> "FM-C: $(…) scope loss. FM-D: ( … ) or & scope loss — export inside BASH_SUBSHELL>0 does not cross back to parent env."

**Current plan state:** §2 M2-0 pre-check correctly states "child processes do not inherit traps unless `set -T` is active." The precondition is documented but not enforced — no gate verifies `set -T` is set.

**Optimization routes surveyed:**

- _Shim-code fix:_ Add `set -T` at top of the shim block (line before `__xbreed_effort_trap()`). Side-effect: all bash functions in this session become trap-visible — potential performance cost for complex prompts. Acceptable for a dev-only session shim.

**Status:** Open — judge call required on whether `set -T` is acceptable given session-wide scope. ## Headline Finding — FM-K: Missing `*)` Default Arm [4-AXIS CONVERGENCE]

**This is the highest-priority structural gap in the shim.**

### What it is

The inner `case "${BASH_REMATCH[1]}"` at `~/.bashrc:239-242` has no `*)` default branch:

```bash
case "${BASH_REMATCH[1]}" in
  cco-*)                                              export CLAUDE_CODE_EFFORT_LEVEL=high ;;
  ccs-distiller-*|ccs-scribe-*|ccs-simplifier-*)      export CLAUDE_CODE_EFFORT_LEVEL=medium ;;
  ccs-*|cdx-*|g-*)                                    export CLAUDE_CODE_EFFORT_LEVEL=high ;;
esac
```

An unrecognised name (e.g., `foo-bar-verify` from M2-1, or any future prefix not yet in the roster) leaves `CLAUDE_CODE_EFFORT_LEVEL` at whatever value it held **from the previous trap fire**. The env var is sticky — it does not reset between spawns.

### Why it matters

The test plan's M2-1 "failing baseline" step (spawn `foo-bar-verify`, expect `UNSET`) is correct only if the tester manually runs `unset CLAUDE_CODE_EFFORT_LEVEL` in the pane immediately before the spawn. The plan requires this:

> "Run this in the pane first to enforce test-order independence: `unset CLAUDE_CODE_EFFORT_LEVEL`"

This workaround is necessary **precisely because there is no `*)` default**. Without the manual unset, M2-1 would return `high` (stale from a prior named spawn), silently passing when it should fail.

**4-axis convergence across 3 model families (Claude opus, Codex, Gemini). Zero dissent.**

### Current shim state

`~/.bashrc:234-247` — documented but NOT PATCHED. The test plan §4 table row for FM-K (last row) acknowledges the bug and describes the fix:

> "Add `*) unset CLAUDE_CODE_EFFORT_LEVEL ;;` to outer case in `__xbreed_effort_trap`. Timing is safe: DEBUG fires before exec, so unset runs in parent before fork; CC child inherits nothing (correct)."

The fix is in the plan's §4 failure-mode table but is labeled as a known issue, not applied. **Open judge call for R2 execution.**

### Proposed patch (minimal)

```bash
case "${BASH_REMATCH[1]}" in
  cco-*)                                              export CLAUDE_CODE_EFFORT_LEVEL=high ;;
  ccs-distiller-*|ccs-scribe-*|ccs-simplifier-*)      export CLAUDE_CODE_EFFORT_LEVEL=medium ;;
  ccs-*|cdx-*|g-*)                                    export CLAUDE_CODE_EFFORT_LEVEL=high ;;
  *)                                                  unset CLAUDE_CODE_EFFORT_LEVEL ;;
esac
```

This makes M2-1 test-order-independent (the manual `unset` pre-step becomes redundant but harmless) and eliminates the sticky-export contamination class entirely.
