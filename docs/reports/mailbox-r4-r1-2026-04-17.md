## 4. Per-finding detail

### M1 — PID-recycle mtime-floor orphan guard (SHIPPED)

**Claim.** `collect_compact_sidecars` orphan-recovery path (`src/mailbox.rs`, R3 Gate 4 landing `24b1538`) checked only `!pid_is_alive(pid)` to decide whether to adopt an orphan `events.compact.<pid>` file. If dead PID A's orphan file exists and a new process B reuses PID A before drain runs, `pid_is_alive(A)` returns true (B is alive) and the orphan is skipped permanently — silent data loss. This is the PID-recycle gap flagged in R3 handoff §R4 candidates.

**Confidence.** HIGH (shipped; green test suite; 3-prefix convergence: ccs-executor-m1 + cco-planner + cdx-reviewer).

**Evidence verbatim.**
- `src/mailbox.rs:205-207`: OR-clause guard: `!pid_is_alive(pid) || orphan_file_has_stale_mtime(&source, Duration::from_secs(60))`
- Helper `orphan_file_has_stale_mtime` at `src/mailbox.rs:250-261`: reads `path.metadata().modified()`, computes `SystemTime::now().duration_since(modified)`, returns `age >= mtime_floor`.
- Test `drain_adopts_orphan_compact_file_with_recycled_pid_after_60s` — primary boundary: file age ≥60s → adopted despite live PID.
- Test `drain_skips_orphan_compact_file_with_recycled_pid_under_60s` — secondary boundary (59s): file age <60s, live PID → skipped (data NOT lost to incorrect adoption).
- `filetime = "0.2"` in `Cargo.toml:31` [dev-dependencies] (contradicts ccs-simplifier-r4 spoof claim — see §8).
- Cargo test result: RED: 1 failed (pre-fix). GREEN: 19 passed; 74/74 total post-fix; clippy clean; fmt clean.

**Linchpin.** The OR-clause is the load-bearing fix: either condition alone adopts the orphan. `orphan_file_has_stale_mtime` provides the PID-recycle escape hatch — even if PID B is alive, a 60s-old file could not belong to B (a process holding a compact rename open for >60s would itself be anomalous). The 59s boundary test closes the mutation gap: without it, the floor could be deleted and the guard collapses to `pid_is_alive`-only.

**Verdict.** SHIPPED (commit included in current working tree; 74/74 green).

If drain I/O is linear in file size, expected p50 delta: ~88% reduction (hand-calc, unverified). Prereqs: M1 (shipped) + M5 (proposed — prevents collision during cadence-triggered compact).

**Confidence.** LOW (hand-calc only; labrat probe BLOCKED; no empirical gate).

**Linchpin.** M2 is the primary L-axis candidate. If I/O is not linear in file size (e.g., OS page-cache dominates at these sizes on WSL2 ext4), the 88% hand-calc collapses. ### M4 — COMPACT_PENDING counter-leak (HIGH, PROPOSED)

**Claim.** `compact_events` at `src/mailbox.rs:41-49` uses an `is_ok` guard that increments `COMPACT_PENDING` before spawning the worker but decrements only inside the worker body (on `Ok` return). If the worker thread panics (caught by `catch_unwind`) or exits abnormally, the `fetch_sub` decrement is skipped. `__wait_compact_idle` spins on `COMPACT_PENDING > 0`, so a single counter-leak causes permanent deadlock on next `compact_idle` call.

**Confidence.** HIGH (mutation 0/2 killed; cdx-reviewer + cco-connector convergence on fix semantics; 2-prefix convergence).

**Evidence verbatim.**
- `src/mailbox.rs:41-49`: `is_ok` guard path — increment before spawn, decrement inside worker body only.
- Mutation test: 0 of 2 generated mutations killed by existing test suite (counter-leak path not exercised by current tests).
- Fix: unconditional `fetch_sub` in a `finally`-equivalent pattern (Rust: `defer`-via-drop or explicit decrement in all paths including `catch_unwind` recovery arm).
- cco-connector: "M4-before-M2 serial commit — M4 is a correctness fix that M2's cadence increase makes load-bearing; counter-leak at IT=100 is rare; at higher cadence it becomes probable." Planner and reviewer convergence on M4-before-M2 ordering.
- M4 is orthogonal to M6 (different guard pattern; both in same function; neither masks the other).

**Linchpin.** M4 is load-bearing for M2: bytes-floor cadence triggers compact more aggressively (potentially 10-100× more frequent). At IT=100 with 100k events, the counter-leak is probabilistic (requires panic during compact worker). At bytes-floor cadence with concurrent writes, the leak is near-certain on any compact-path panic. M4 must land before M2 per planner+connector+reviewer consensus.

**Verdict.** PROPOSED (HIGH). 1-line unconditional fetch_sub fix. R2 executor task, serial before M2.

### M5 — same-PID compact collision (HIGH, PROPOSED)

**Claim.** `compact_events` and the sync-fallback path (`src/mailbox.rs:315`, `L378-379`) both use `std::process::id()` as the sole suffix for the compact sidecar filename (`events.compact.<pid>`). If two compact calls occur in the same process (worker thread + sync fallback via channel Full at `L440`), both generate the same filename. The second `rename(2)` clobbers the first sidecar — events silently overwritten.

**Confidence.** HIGH (4-source cross-model convergence: cdx-reviewer + ccs-mutation-r4 + ccs-connector + cco-critic = 4 sources across 3 prefixes; ccs-mutation constant-PID kill evidence narrows but does not eliminate the race).

**Evidence verbatim.**
- `src/mailbox.rs:315`: `std::process::id()` suffix in compact sidecar rename.
- `src/mailbox.rs:378-379`: sync fallback path uses same `std::process::id()` suffix.
- `src/mailbox.rs:440`: channel `Full` error triggers sync fallback — the collision window.
- Mutation-tester: constant-PID mutations (Scenario 1 — rename exclusion) 0/2 killed by test suite. rename(2) on Linux is atomic but not exclusive — second rename overwrites the first sidecar file atomically. Scenario 2 (channel Full + concurrent writes): real race, no rename exclusion.
- cdx-reviewer initial severity: BLOCKER. Downgraded to HIGH after mutation evidence showed Scenario 1 (concurrent drains racing the same sidecar) is partially mitigated by rename exclusion in `collect_compact_sidecars`. Scenario 2 (sync fallback clobber) remains a live HIGH race.
- Fix: static `AtomicU64` seq suffix — increment per compact call. Guarantees unique filename within a process across worker + sync fallback paths.

**Linchpin.** Scenario 2 is the live path: `sync_channel(1)` cap=1 (R3 Gate 1) means any second concurrent compact call in the same process hits `channel Full` and takes the sync fallback path. Both paths then race to write `events.compact.<pid>` — same filename, one wins silently. At bytes-floor cadence (M2) with concurrent writes, the collision probability is non-negligible. M5 must land before M2.

**Verdict.** PROPOSED (HIGH). ~5 LOC fix (AtomicU64 seq suffix). R2 executor task. Red-before-green gate required.

### M6 — IO-error silent swallow (HIGH, PROPOSED)

**Claim.** `compact_events` at `src/mailbox.rs:41` uses `result.is_ok()` to check the `JoinHandle` result. `JoinHandle::join()` returns `Ok(thread_result)` when the thread exits without panic; the inner `thread_result` is the worker's own `Result<(), io::Error>`. `is_ok()` on the outer `Ok` is always true regardless of whether the inner result is `Ok(Ok(_))` or `Ok(Err(_))` — an I/O error inside the worker (e.g., sidecar write failure, rename failure) returns `Ok(Err(e))`, which `.is_ok()` evaluates as true. The error is silently discarded.

**Confidence.** HIGH (cdx-reviewer surfaced; cco-connector confirmed; orthogonal to M4; 2-prefix convergence).

**Evidence verbatim.**
- `src/mailbox.rs:41`: `result.is_ok()` — evaluates the outer `JoinHandle` result, not the inner worker result.
- Pattern: `Ok(Ok(_))` → true; `Ok(Err(e))` → also true (error discarded); `Err(panic)` → false.
- Fix: explicit match on the nested result — `Ok(Ok(_))` succeeds; `Ok(Err(e))` logs error + propagates; `Err(panic)` handled by existing `catch_unwind`.
- Orthogonal to M4: M4 is about the counter decrement path; M6 is about the result inspection path. Both can land independently; no ordering constraint between M4 and M6.

**Linchpin.** Silent I/O error swallow means compact failures are invisible to the caller and to monitoring. At bytes-floor cadence (M2), compact runs more frequently — any recurring I/O error (disk full, permission change, sidecar path collision) would be silently ignored, causing the live mailbox to grow unbounded without triggering any alert path.

**Verdict.** PROPOSED (HIGH). 2-line explicit match fix. R2 executor task. Orthogonal to M4.

### CRITIC-HALT-RETHINK — methodology carry (R2 gate redesign required)

**Source.** cco-critic-r4 late arrival. Heuer-ACH critique of the standing HALT rule.

**Claim.** The HALT rule `p50 delta < 5%` is below the noise floor of the current bench harness. At baseline variance 2.93% (ITERATIONS=100, `docs/reports/mailbox-bench-baseline.json`) and p50 = 41.25ms, the one-sigma noise band is ±1.21ms = ±2.93%. A 5% delta = 2.06ms falls within 1.7σ of the noise distribution. At SNR≈1.7, the HALT rule fires on noise roughly 9% of the time (one-tailed), rejecting real wins as noise and accepting noise as wins.

**Confidence.** HIGH (cco-critic Heuer-ACH framework; arithmetic is primary-source verifiable from `mailbox-bench-baseline.json`; no empirical contradiction found).

**Evidence verbatim.**
- Baseline: p50 = 41.25ms, variance 2.93% (3 consecutive runs @ ITERATIONS=100).
- One-sigma noise: 41.25ms × 0.0293 = 1.21ms.
- HALT threshold: 41.25ms × 0.05 = 2.06ms.
- SNR = 2.06 / 1.21 = 1.70.
- At SNR=1.7, HALT rule fires on noise ~9% of runs (z=1.7, one-tailed p≈0.045; two-tailed p≈0.09).
- Proposed replacement: **paired** bootstrap 95% CI of per-iteration delta excludes zero (candidate < baseline). Per cco-critic-r2 R2 reframe: unpaired CI inflates dispersion on non-i.i.d. warm-cache samples; paired-delta CI captures the correct uncertainty. `pooled_SD` is dropped as a wrong-scale heuristic for median-delta CI (R2 critic flagged unpaired OR-gate as admitting noise-wins).
- Secondary critic proposals (convergent with reviewer/connector/mutation):
  - M4-before-M2 serial commit ordering.
  - M3 acceptance gate: `p95 > 30ms` (explicit threshold, not % delta).
- xask codex for labrat-cadence redo: BLOCKED (codex-spark timeout 540s × 2 attempts).

**Linchpin.** If the HALT rule is below the noise floor, M2's hand-calc 88% delta would clear any reasonable gate — but the current rule cannot certify it. The CI-based gate is a prerequisite for trusting R2 bench results on M2.

**Verdict.** R2 CARRY. Switch HALT rule to CI-based gate. Document in `benches/mailbox.rs` before M2 bench run.

## 5. Rejected alternatives survey

### M3 fallback-to-full (flat, rejected as primary)

**Probe.** Incremental drain as a primary (non-conditional) strategy: always drain the N most recent events and leave the tail.

**Result.** Blocked by semantic constraint: drain-entry trigger only, never mid-stream. A partial drain that returns a subset of events violates the "one drain = complete current mailbox" invariant used by hook injection callers. Fallback-to-full p50 is 41–44ms (flat vs baseline) — no regression, but also no win. Incremental drain as primary is rejected; retained only as M2-conditional fallback.

**Not retried.** Streaming-incremental drain (read N lines then suspend) requires an API change to `drain_events` signature. Deferred to R5 if M2+M3 combined still miss the L-axis gate.

### Scout growth_ratio dropped → bytes-floor only (connector YAGNI critique, accepted)

**Original scout position.** `file_bytes >= FLOOR AND growth_ratio >= 1.5×` — compact only when the file is both large enough and growing faster than 1.5× since last compact. Two-parameter gate.
