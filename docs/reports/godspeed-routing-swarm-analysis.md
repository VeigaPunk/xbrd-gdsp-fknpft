# Godspeed Analysis: Codex Routing & Swarm

1. **scripts/xask 4-layer transport gate**: No tmpdir isolation used for codex (unlike gemini rich mode). Safe from tmpdir collisions, but vulnerable to env var bleed if swarmed within the same process.
2. **src/guard.rs policy deny-list**: The deny-list sees `Bash` tool with the command string `xask ...`. Changing the model from `gpt-5.4` to `gpt-5.4-mini` does not alter the command string evaluated by the policy. The deny-list is blind to the routing change.
3. **src/ask.rs timeout + retry logic**: `XASK_TIMEOUT_SECS` is 300s. The outer orchestrator `xask codex` will wrap the 10 inner `xask --spark codex` subprocesses. If the outer process starts with `XASK_TIMEOUT_SECS=300`, and passes this down via env vars, the 10 inner processes will inherit it. If the swarm runs concurrently (as now documented: `xask codex "Orchestrate 10 parallel..."`), it fits within the 300s window. 
4. **agent templates / shared.md**: `xask -R codex` references now correctly indicate `gpt-5.4-mini` instead of `full gpt-5.4`. The labrat swarm documentation incorrectly used `xask --spark codex` as the orchestrator instead of `xask codex`; this was fixed.
5. **test harness**: `tests/ask_with_loadout.rs` and `src/ask.rs` assertions were updated to reflect that both `-R` and default lanes now utilize `gpt-5.4-mini`.
6. **Second-order swarm regressions (Whole-table)**: A hidden race condition existed in `src/mailbox.rs` `collect_compact_sidecars`. Concurrently draining the mailbox caused duplicate reads of sidecar events because the `rename` target (`events.compact_ready.drained_by.PID.SEQ`) shared the `sidecar_prefix`. This caused subsequent concurrent thread `read_dir`s to re-parse the already renamed file and re-rename it ad infinitum, breaking under a swarm load. This race was identified and surgically fixed.
