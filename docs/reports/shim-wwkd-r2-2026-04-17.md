## Findings

### I1 — Witness-leak: `XBREED_EFFORT_SHIM_HIT` cleared before parent can read it [RETHINK]

**Source:** cco-critic-r2-postpatch (opus 4.7 high, heuer-planning). Primary-source verified `~/.bashrc:255-258`.

**Mechanism:** The outer `*)` arm fires on EVERY `$BASH_COMMAND` that does not match the glob. This includes the very next command after a spawn — `echo`, `printf`, `:`, any shell built-in. The sequence in the parent shell:

```
1. : --agent-name ccs-scribe-test --team-name t
   → DEBUG fires → outer glob matches → inner case → EFFORT=medium, HIT=ccs-scribe-test:medium:<pid>
   → exec: CC child forks, inherits env snapshot (both vars present) ✓

2. echo "${XBREED_EFFORT_SHIM_HIT}"          # parent tries to read HIT
   → DEBUG fires BEFORE echo executes → outer *) matches → unset EFFORT, unset HIT
   → echo executes → prints empty string    ✗
```

**Child-side is correct:** The CC child captured the env at fork time (step 1). HIT is present inside the spawned teammate's env. Test steps that check HIT inside the labrat (via `env | grep XBREED_EFFORT_SHIM_HIT` in the prompt) work correctly. Parent-side `echo` or `printf` checks are structurally broken.

**Affected test plan step:** M2-6, line:
```bash
echo "witness: ${XBREED_EFFORT_SHIM_HIT}"
# Expected: ccs-labrat-rev:high:<BASHPID>
```
This will ALWAYS print `witness: ` (empty). The `echo` command fires the `*)` arm before it executes.

**Fix:** Remove parent-side HIT check from M2-6. Replace with spawn-side check: add `printf '%s' "$XBREED_EFFORT_SHIM_HIT"` to labrat prompt and SendMessage. Confirmed pattern already used correctly in M2-3 (env check inside labrat).

**Mitigation rejected:** Moving `unset XBREED_EFFORT_SHIM_HIT` out of outer `*)` (keep only `unset CLAUDE_CODE_EFFORT_LEVEL` there) — rejected. The witness var should be cleaned on every non-spawn command for the same stale-contamination reason as the effort var. Parent-side verification is not the design goal; child-side is. No fix to shim code needed for I1.

### I2 — Inner case missing `*)` fallback for unmapped prefixes [RETHINK — BLOCKS GATE]

**Source:** cco-critic-r2-postpatch. Primary-source verified `~/.bashrc:239-252`.

**Mechanism:** The inner `case "${BASH_REMATCH[1]}"` covers `cco-*`, `ccs-distiller-*|ccs-scribe-*|ccs-simplifier-*`, and `ccs-*|cdx-*|g-*`. Agent names that match the outer glob but not any inner arm — `the-planner`, `the-judge`, `sentinel`, `reviewer`, `distiller` (without `ccs-` prefix), any future unregistered prefix — fall through silently:

```
outer glob: MATCHES (has --agent-name + --team-name)
inner regex: MATCHES (valid charset, captures name)
agent="the-planner"
inner case: no arm matches → case exits, CLAUDE_CODE_EFFORT_LEVEL UNCHANGED
            XBREED_EFFORT_SHIM_HIT NOT SET
fi → outer arm exits
exec: CC child inherits UNSET for both (correct effort: no tier assigned)
     BUT: cannot distinguish from "shim never fired at all"
```

**Why it matters:** When the judge dispatches `the-planner` or `the-judge` teammates (which appear in real xbgst runs), the shim is silent — no witness, no tier. If a bug prevents the trap from firing at all, the symptom is identical. The `XBREED_EFFORT_SHIM_HIT` witness var, added in R2 specifically for this discrimination, fails for unmapped names.

**Proposed one-line fix (Task A):**
```bash
case "${BASH_REMATCH[1]}" in
  cco-*)
    export CLAUDE_CODE_EFFORT_LEVEL=high
    export XBREED_EFFORT_SHIM_HIT="$agent:high:$BASHPID"
    ;;
  ccs-distiller-*|ccs-scribe-*|ccs-simplifier-*)
    export CLAUDE_CODE_EFFORT_LEVEL=medium
    export XBREED_EFFORT_SHIM_HIT="$agent:medium:$BASHPID"
    ;;
  ccs-*|cdx-*|g-*)
    export CLAUDE_CODE_EFFORT_LEVEL=high
    export XBREED_EFFORT_SHIM_HIT="$agent:high:$BASHPID"
    ;;
  *)
    export XBREED_EFFORT_SHIM_HIT="$agent:NOMATCH:$BASHPID"
    ;;   # CLAUDE_CODE_EFFORT_LEVEL left unset — correct, outer *) will clear on next cmd
esac
```

Note: do NOT `unset CLAUDE_CODE_EFFORT_LEVEL` in the inner `*)` — the outer `*)` handles all non-spawn commands. The inner `*)` fires inside the `if [[ BASH_COMMAND =~ ... ]]` block, meaning the command IS a spawn but with an unrecognized name. The effort var should be unset (no tier assigned); this is already handled because EFFORT was unset by the PRIOR outer `*)` fire (the pre-spawn commands cleaned it). Adding an explicit `unset` here is redundant but not harmful.

**Gate impact:** Until this patch is applied, the R2 gate's `after-unmapped` line will show `HIT=UNSET` instead of `HIT=the-planner:NOMATCH:<pid>`. Gate BLOCKED.

### I3 — BASHPID semantics in witness var [CONSIDER]

**Source:** cco-critic-r2-postpatch. Non-blocking.

**Issue:** `XBREED_EFFORT_SHIM_HIT="$agent:$tier:$BASHPID"` — `$BASHPID` is the PID of the interactive bash session that set the var (the parent shell), not the spawned CC child's PID. A reader of the var inside the spawned teammate sees the parent's PID, which may be confusing ("this PID is not me").

**What `$BASHPID` is useful for:** Identifies which shell session wrote the witness — useful if multiple tmux panes have the shim active and spawn teammates concurrently. It's a "set-by" PID, not a "self" PID.

**Alternatives surveyed:**
- `$BASHPID` → rename to encode intent: `"$agent:$tier:parent=$BASHPID"` — more explicit but changes the format consumed by any tooling that parses the var.
- Replace with `$EPOCHREALTIME` (bash 5.0+) — makes the witness a timestamp, useful for debugging trap-fire ordering.
- Keep `$BASHPID` as-is — simplest; add a comment in the shim.

**Recommendation:** Keep `$BASHPID`. Add a one-line comment above `XBREED_EFFORT_SHIM_HIT` export in the shim:
```bash
# BASHPID = this interactive shell's pid (set-by, not child pid)
```

Non-blocking. Deferred to R3 or a standalone shim cleanup commit.
