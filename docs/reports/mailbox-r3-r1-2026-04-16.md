## Per-finding detail

### move_001 — collect_compact_sidecars: 4 surviving mutation vectors CLOSED (ACCEPT, escalated HIGH)

**Claim.** `collect_compact_sidecars` (`src/mailbox.rs:96–142`) had four untested behavioral branches, each a surviving mutation: (1) the `.tmp` suffix filter at L121 that excludes in-progress sidecar writes, (2) the exclusive-rename claim at L132 (`std::fs::rename(&sidecar, &drain_sidecar)`), (3) the concurrent-drain double-count prevention (two drains racing for the same sidecar — one must win, the other silently skip), and (4) the empty-mailbox path that still reads sidecars (L61–62 early-return to `collect_compact_sidecars`).

**Confidence.** HIGH (escalated from MEDIUM by distiller). Same-prefix cap note: ccs-mutation-tester and ccs-executor-core both assessed this function — shared `ccs-` prefix caps individual confidence at MEDIUM; escalation to HIGH approved by judge based on mutation evidence weight. move_002 (identical scope, same kill test) absorbed before escalation, preventing double-count inflation.

**Evidence (verbatim from executor + test code).**
- Commit `f189d43`: `test(mailbox): R3 M-tests — cover concurrent-drain ownership + .tmp sidecar filter`
- Test 1: `compact_sidecar_consumed_exactly_once_under_concurrent_drain` (`src/mailbox.rs:534–572`) — places a pre-built sidecar (`events.compact_ready.99999`) with 10 events; two drain threads race on `Barrier::new(2)`; asserts `r1.len() + r2.len() == 10` (exclusive ownership, no double-count).
- Test 2: `drain_skips_compact_ready_tmp_sidecars` (`src/mailbox.rs:575–605`) — places a `.tmp` sidecar (ghost, must be ignored) and a real sidecar (must be consumed); asserts `events.len() == 1` and `tmp_sidecar.exists()` after drain.
- Both tests pass in `cargo test --lib mailbox` (13/13 → 15/15 post-commit).

**Linchpin.** L121 `name_str.ends_with(".tmp")` and L132 `std::fs::rename` are the two load-bearing mutations. Without the `.tmp` guard, in-progress compact writes would be consumed mid-write, corrupting sidecar content. Without the rename claim, two concurrent drains could both `read_to_string` the same sidecar and return duplicate events — violating the I7 fix's "no silent loss, no double-count" invariant.

**Verdict.** ACCEPT (HIGH). Tests shipped at `f189d43`.

**Rationale.** Double-count of sidecar events would silently inject duplicate events into the next hook injection, breaking the "one event == one NDJSON line" invariant (I1). The `.tmp` filter failure would corrupt events mid-compact-write. Both are safety-critical for the sidecar pattern that closed I7.

### move_003 — Option E (idle-thread compact defer) ACCEPT — reframed as CLI-ergonomics

**Claim.** `compact_events` currently blocks the caller thread for ~24.5ms p50 (n=100k, `docs/reports/mailbox-bench-baseline.json`). Deferring compact to an idle background thread (Redis AOF `no-appendfsync-on-rewrite` pattern, SQLite WAL `wal_autocheckpoint(0)` pattern) would eliminate this from the caller's measured latency entirely.

**Confidence.** MEDIUM (architectural; no bench delta yet — implementation pending R2). Same-prefix cap: cco-the-planner-r3 (cco-) is capped at independent MEDIUM for architectural projection without empirical gate.

**Evidence verbatim.** `docs/reports/mailbox-bench-baseline.json` → `function_wall_ms.compact_events.n=100000.p50_ms: 24.499337`. Planner addendum-3 (Redis AOF prior art): "no-appendfsync-on-rewrite decouples compaction from write path; caller-invisible during rewrite via in-memory accumulation buffer — direct prior art for Option E." Planner addendum-9 (R4 carry-forward): "Option E unlocks compact-cadence aggressiveness; when compact is off-thread, the cadence gate (how often to compact) can be tuned without affecting caller latency, reframing the CPU-vs-I/O tradeoff for R4."

**Linchpin.** The current sidecar implementation (R2, commit `1463936`) already structurally decouples compact from the live mailbox write path — compact writes to `events.compact_ready.<pid>` and never touches `path` after initial rename. Option E adds caller-invisibility on top of this foundation: the compact thread runs only when idle (no writers detected), so the caller's drain latency never includes compact wall-time.

**CONTRADICTION-3 resolution.** Connector raised: "is compact on the hot path or CLI-only?" Critic (move_009) reframed: compact is on the *caller's* path (the CLI process that runs `xbreed compact`) but NOT on the write_event hot path. Judge resolution: Option E is correctly scoped as a CLI-ergonomics win — user-facing latency reduction when running `xbreed compact`, not a write_event throughput improvement. The reframe is accurate and the axis label remains A-decouple.

**Verdict.** ACCEPT — R2 implementation scope.

### move_004 — parse-layer CPU axis: all 4 arms CLOSED (REJECT)

**Claim.** The CPU-parse bottleneck in `drain_events` (serde_json per-line deserialization, ~40ms of the 42ms p50 at n=100k per planner initial framing) is addressable via simd-json or zero-copy Cow/SmolStr field variants.

**Confidence.** HIGH (empirical rejection, multi-arm). All 4 arms measured; all fail the "both p50 AND p95 improve" gate.

**Evidence (verbatim ratios from labrat probes).**

| Arm | Probe | Result ratio (candidate/baseline) | Additional defect |
|-----|-------|------------------------------------|-------------------|
| M3-simd | simd-json per-line | **0.57×** (43% slower) | padded-buffer copy overhead; no size win |

After M4b showed +64% alloc bytes despite the Cow<'de, str> optimization (unexpected: serde_json_borrow allocates interning structures that dominate at n=100k NDJSON), the planner issued addendum-6 retracting the alloc-bound decomposition: "M4 empirical refutes the alloc-bound framing; the drain bottleneck is not allocation — it is the serde parse evaluation loop itself, which neither SmolStr nor Cow eliminates."

**Verdict.** CLOSED. No productive parse-axis path on this workload at WSL2 ext4 n=100k. Not a R4 candidate unless a new probe design avoids the per-line boundary overhead (e.g., bulk NDJSON streaming with simd-json's streaming parser).

## Rejected alternatives survey

### M3-simd (0.57× baseline)

**Probe.** simd-json per-line: for each line from the drain file, call `simd_json::from_slice_borrowed` on a padded `AlignedVec` copy of the line bytes.

**Result.** 0.57× baseline p50 at n=100k — 43% regression. The padded-buffer copy per line negates the AVX2 parse speedup: at average NDJSON line length ~75 bytes, the `AlignedVec` allocation and memcpy overhead exceeds the simd parse savings. simd-json's structural advantage materializes at bulk document sizes (≥4KB per parse call), not per-line slices.

**Not retried.** Bulk simd-json parse (one call for full file content) is a different architecture requiring structural NDJSON changes (no per-line deserialization). Deferred; not a R4 candidate without re-design.

---

### M3-alt (0.53× + correctness kill on escaped quotes)

**Probe.** Alternative simd-json calling convention: `simd_json::from_str_unchecked` after UTF-8 validation, reusing the existing `read_to_string` buffer.

**Result.** 0.53× baseline p50 (47% regression) + correctness kill: NDJSON lines containing `\"` (escaped double-quote in payload field) were corrupted by simd-json's in-place mutation of the input buffer. `from_str_unchecked` modifies the `&mut str` in-place for performance; after the call, the string contains simd-json's tape encoding, not valid UTF-8. Lines parsed correctly on first read but second-parse validation (used in the test harness) failed.

**Verdict.** Double-disqualified: performance regression + correctness kill. Closed.

---

### M4b Cow/BorrowedEvent (0.38× + 64% alloc bytes)

**Probe.** Replace `Event { from: String, event_type: String, payload: String }` with `BorrowedEvent<'de> { from: Cow<'de, str>, event_type: Cow<'de, str>, payload: Cow<'de, str> }` using `serde_json_borrow`.

**Result.** 0.38× baseline p50 (62% slower) + +64% total alloc bytes per drain at n=100k. The alloc regression is explained by `serde_json_borrow`'s interning structures (a `BorrowedValue` arena is allocated per document) which dominate at n=100k × 75-byte lines. The borrowed references provide zero benefit when the events are immediately collected into a `Vec<Event>` (which clones the borrowed strs to owned Strings anyway, per `Event` struct's owned fields).

**Planner retraction.** Planner addendum-1 claim "alloc-bound at 40ms" invalidated by this result. Addendum-6 correction: "alloc is not the bottleneck; parse evaluation loop is."

**Verdict.** Rejected. Lifetime parameter ripple through 4 callers (`format_hook_injection`, `src/main.rs:77–98`, test helpers, bench seeders) adds refactor cost with negative empirical return.

---

### M4a SmolStr (0.89×, rejected R2 labrat sweep)

**Probe.** Replace `String` fields in `Event` with `SmolStr` (inline storage ≤23 bytes; heap-spills for larger). Hypothesis: event_type and from fields are typically short-lived labels that fit inline.

**Result.** 0.89× baseline p50 (11% slower). SmolStr's inline-23 boundary is frequently exceeded by payload field (average payload length at n=100k bench: 68 bytes). The heap-spill path on every payload event negates the inline gains on short fields. Net: marginal deficit.

**Verdict.** Rejected (insufficient gain, non-zero cost). Not a R4 candidate.

---

### M5 H12 persistent fd (TOCCTOU amplifies best-effort window)

**Probe.** Not implemented (blocked at design stage; see move_006).

**TOCCTOU amplification argument.** If a writer holds a persistent fd open across a drain's rename, the writer's next `write_all` targets the *drained* inode (which is now orphaned post-rename, pending deletion). On Linux ext4, writes to an orphaned inode succeed at the kernel level (the inode is still referenced by the fd) but the data is not accessible via the filesystem path — it will be GC'd when the fd closes. This silently discards events: no error, no eprintln, the write_all returns Ok(()). The best-effort window documented at `src/mailbox.rs:51–56` (accepted trade-off for the `rename → read` drain) becomes a deterministic data-loss path under H12.

**Verdict.** Informational blocker. Do not implement until Option E architecture provides a quiescence gate that prevents writes during compact + provides fd invalidation semantics.

## R4 carry-forward (planner addendum-9)

Option E (accepted for R2 implementation) unlocks two R4 opportunities that are currently blocked by the synchronous compact architecture:

1. **Compact-cadence aggressiveness.** When compact runs off-thread, the cadence decision (how often to compact) is decoupled from caller latency. R4 can tune cadence aggressively (e.g., compact after every 1k events) without penalizing the CLI user. Current constraint: compact at ~24.5ms p50 makes frequent invocation unacceptable for interactive use.

2. **CPU-vs-I/O latency leverage reframe.** After Option E, the caller's observed latency is dominated by `drain_events` (42ms p50 at n=100k) with zero compact contribution. The R4 parse-axis question becomes: "can we cut drain's 42ms p50?" without the compact-axis confound. If a new parse architecture (bulk simd-json streaming) or a new drain strategy (incremental drain, streaming instead of read-all) shows a p50 win ≥10%, R4 has a clear gate to accept it.
